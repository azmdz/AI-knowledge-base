# 設計ドキュメント

## 0. プロダクト哲学

### 行動指針：Done is better than perfect

このサービスは「完璧な記事より、公開された記事」というひとつの行動指針を中心に設計されています。

**狙うペルソナ**

「書き殴った初稿はある。でも、このままでは公開できない」という状態の人です。知識はある。書く意欲もある。しかし「クオリティが不安」という心理的障壁で公開に至らない。

このサービスのゴールは、その人を **Draft → Published** まで連れていくことです。

---

### なぜ汎用 AI（ChatGPT 等）では代替されないか

「ChatGPT に貼り付けて整えればいいのでは？」という疑問は正当ですが、このサービスが汎用AIと異なる点は以下の3つです。

| 比較軸 | 汎用 AI | このサービス |
|--------|---------|------------|
| プロンプト | 毎回ユーザーが考える | 社内文書向けに調整済み・メンテされる |
| コンテキスト | 一般的な文書標準 | 社内記事データを参照した校正 |
| 品質判断 | ユーザーが「これでいいか？」を自分で判断 | サービスが社内水準を担保。ユーザーは事実確認のみ |

品質判断をサービス側が引き受けることで、ユーザーは「完璧かどうか」ではなく「事実として正しいか」だけを確認すれば公開できます。

---

### 正のフィードバックループ

公開ハードルを下げることは記事を増やすだけでなく、サービス全体を正のループへ転換します。

```
公開ハードルを下げる（AI校正）
        ↓
   記事が増える
        ↓
  検索精度が上がる
        ↓
利用者が答えを見つけられる
        ↓
  成功体験が生まれる
        ↓
また書きたくなり、記事が増える ──→（繰り返し）
```

このループが回り始めると、ナレッジベースが自律的に成長する状態になります。

## 1. 課題分析

### 1.1 問題の本質：4つの症状ではなく1つの負のループ

「利用率が低い・投稿されない・検索されない・定着しない」は独立した4問題ではなく、**相互依存する負のフィードバックループ**です。

```
記事を書く摩擦が高い
        ↓
  投稿されない
        ↓
   記事が少ない
        ↓
検索しても役に立たない
        ↓
    使われない
        ↓
投稿する動機が生まれない ──→ (最初に戻る)
```

UI改善や周知メールはループの一部を緩和しますが、構造を変えない限り同じ状態に戻ります。

### 1.2 Fogg 行動モデル (B = M × A × T) による分解

| 障壁 | 種別 | コードで解決できるか | 深刻度 |
|------|------|---------------------|--------|
| 検索しても答えが見つからない | 能力(A)の欠如 — 読み手 | ✅ | ★★★★★ |
| 記事の書き方・品質水準がわからない | 能力(A)の欠如 — 書き手 | ✅ | ★★★★☆ |
| 記事がそもそも少ない（コールドスタート） | トリガー(T)の欠如 | △（組織施策が主、シードデータはデモ用） | ★★★★☆ |
| 投稿してもフィードバックがない | 動機(M)の欠如 | △ | ★★★☆☆ |
| 文化・運用オーナー不在 | 環境 | ❌ | ★★★★★ |

### 1.3 最大レバレッジの選定

コードで解決できるボトルネックの中から、**「書く摩擦」** を最優先にします。

記事が増えなければ検索成功率はそもそも動きません。供給（投稿）を増やすことが、検索精度の向上・利用定着のすべての前提です。そのうえで、増えた記事を読み手が使えるようにするために **「検索の失敗」** を並行して解消します。

- 書く摩擦の解消（**北極星**） → 投稿数が増える → 記事が蓄積される
- 検索の失敗を解消（遅行指標） → 読み手がナレッジを得られる → 投稿への動機づけになる

## 2. 解決アプローチ

### 2.1 読み手側：AI Q&A 検索（RAG）

従来のキーワード検索は「正確な単語を知っていること」を前提とします。社内ナレッジの読み手は多くの場合、用語が曖昧なまま検索します。

**実装方針**

1. 記事投稿時に本文をベクトル化し `embedding` カラムへ保存
2. 検索クエリを同じモデルでベクトル化し、コサイン類似度で上位 k 件を取得
3. 上位記事をコンテキストとして LLM に渡し、引用付きの自然言語回答を生成（RAG）
4. 「役に立ちましたか？」フィードバックで品質を継続測定

**埋め込みモデルの選択**
Anthropic API には埋め込みエンドポイントがないため、日本語（CJK文字バイグラム）+ 英語（英数字トークン）対応の自前 TF-IDF 256次元ベクトルを使用します。Claude / OpenAI / Mock の全プロバイダーで同一の実装を使うため、シードデータとの次元が一致します。

### 2.2 書き手側：AI 支援による「Draft → Published」の実現

「Done is better than perfect」という行動指針のもと、**初稿から公開完了まで**のステップを最小化します。

ターゲットは「書き殴った内容はあるが、クオリティ不安で公開できていない」状態の人です。AI に品質判断を委ねることで、ユーザーは「これは事実として正しいか？」だけを確認すれば公開できます。

**設計上の重要な決定**

| 決定 | 理由 |
|------|------|
| `bodyRaw`（原文）を常に保持 | 「人間が書き、AIが補助した」という事実を残す。AIが主著者になることを防ぐ |
| AI整形・タグ提案を分離 | 整形だけしたい・タグだけ見直したい、という部分的な利用に対応するため |
| AI整形結果は人間が承認後のみ `bodyFormatted` へ保存 | AI出力の自動公開は誤情報リスクがあるため、必ず人間のレビューを挟む |
| AI編集をgit-like diffで表示 | 変更箇所を一目で確認でき、採用/却下の判断コストを下げる |
| 編集採用/却下をログに記録 | AIが実際に役立っているかを定量的に検証するため |

**推奨フロー（最速パス）**

```
[メモ・下書きをそのまま貼り付ける]
        ↓
[「AI整形」ボタン → 結果を確認 → 採用]
        ↓
[「AIタグ提案」ボタン → 使えるタグをクリック]
        ↓
[「公開する」]   ← ここまで数分
```

**AI編集指示フロー（細かく調整したい場合）**

```
[AI編集指示を入力] → POST /api/ai/edit
        ↓
[差分ビューで確認]
        ↓
[採用] → 本文を更新 + EditLog(accepted=true)
[却下] → 変更なし + EditLog(accepted=false)
```

### 2.3 コールドスタート問題

コールドスタート問題には2つの層があります。**技術的な層**（記事ゼロではデモが動かない）と**組織的な層**（誰も最初に書かない）です。両方を手当てしないと正のループは起動しません。

#### 技術的コールドスタート：シードデータ

15件のシードデータを `prisma/seed.ts` で投入し、初回デモが即座に成立する状態を作ります。検索・AI Q&A・タグフィルタがすべて動作する状態でデモできることが、導入説明の説得力に直結します。

#### 組織的コールドスタート：ファーストペンギンを作る

シードデータは「デモが動く」状態を作るだけで、本物のコールドスタート（誰も書かない問題）は解決しません。この層の解決はツールではなく運用施策です。

| 施策 | 内容 | 効果 |
|------|------|------|
| **導入アナウンス** | Slack 等で「書くだけで AI が整えてくれる」体験を告知 | 認知の獲得。ただし行動に移すコストはユーザー任せ |
| **ハンズオンワークショップ** | 「下書きを持ってきて、その場で公開まで」形式で実施 | Done is better than perfect の体験を直接作れる。最初の成功体験が継続動機になる |
| **推薦投稿依頼** | 知識を持つメンバーに個別に「これを記事にしてほしい」と依頼 | 依頼があることがトリガー(T)として機能する |

**推奨順序**: ワークショップ → アナウンス → 推薦依頼。このサービスの価値（AI整形で公開ハードルが下がる）は「使ってみないとわからない」体験依存のため、ワークショップで直接体験させることが最もレバレッジが高い。

### 2.4 フィードバックループ：Reactions

読み手が「いいね / 役立つ」を付けることで書き手に貢献が可視化され、投稿動機（M）を補強します。

- `Reaction.type`: `like` | `helpful` の2種類

## 3. アーキテクチャ

### 3.1 技術スタック

| 層 | 採用技術 | 選定理由 |
|----|---------|---------|
| フレームワーク | Next.js 16 App Router | サーバーコンポーネント・APIルートを1リポジトリで管理できる |
| 言語 | TypeScript | 型安全性による品質担保 |
| スタイル | Tailwind CSS v4 | ユーティリティファーストで高速なUI実装 |
| ORM | Prisma 7 + libSQL Driver Adapter | SQLite をそのまま使いつつ将来的に PostgreSQL へ移行可能 |
| バリデーション | Zod | API境界での入力検証をスキーマとして表現 |
| エディタ | Tiptap | WYSIWYG × Markdown の両立（Notion風のショートカット対応） |
| 差分表示 | diff (npm) | 行単位の差分計算 |
| AI | Anthropic / OpenAI / Mock | 環境変数で切り替え可能な抽象層 |

### 3.2 AIProvider 抽象化

AIプロバイダーをインターフェースで抽象化し、環境変数 `AI_PROVIDER` で実装を切り替えます。

```typescript
interface AIProvider {
  format(raw: string): Promise<FormatResult>                      // 記事整形（本文・タイトルのみ）
  suggestTags(body: string): Promise<string[]>                    // タグ提案（独立操作）
  edit(body: string, instruction: string): Promise<string>        // AI編集
  embed(text: string): Promise<number[]>                          // ベクトル化
  answer(query: string, context: ArticleContext[]): Promise<SearchResult>  // Q&A
}

// FormatResult: { title: string; body: string }  ※ tags は含まない
```

| `AI_PROVIDER` | 実装クラス | 用途 |
|---------------|-----------|------|
| `anthropic` | `RealProvider` | 本番・デモ（`claude-opus-4-8`） |
| `openai` | `OpenAIProvider` | OpenAI API 利用時（`gpt-4o`） |
| `mock` | `MockProvider` | APIキー不要の開発・テスト用 |

選択順序：`openai` → `anthropic` → `mock`（APIキーが存在しない場合は自動フォールバック）

### 3.3 ユーザー管理

MVP では認証を実装せず、シードユーザー（3名）のヘッダー切り替えで代替します。

```
UserProvider (lib/user-context.tsx)
  ↓ GET /api/users からユーザー一覧をフェッチ
  ↓ 選択を localStorage に永続化
  ↓ useCurrentUser() で全コンポーネントが参照
```

- 記事投稿時の `authorId`
- Reactions の `userId`
- 下書き一覧のフィルタ

どれもこの `currentUser` を参照します。

### 3.4 レート制限

AI 系エンドポイントの意図しない高コスト呼び出しを防ぐため、インメモリレートリミッター（`lib/rate-limit.ts`）を実装しています。

| エンドポイント | 上限 |
|--------------|------|
| `POST /api/ai/format` | 10回/分/IP |
| `POST /api/ai/tags` | 10回/分/IP |
| `POST /api/ai/edit` | 10回/分/IP |
| `POST /api/search` | 20回/分/IP |

> **注意**: インメモリ実装のため、複数インスタンスにスケールアウトした場合はインスタンスをまたいだカウントができません。本番化時は Redis/Upstash 等に置き換えてください。

### 3.5 データモデル

```
User ──< Article ──< ArticleTag >── Tag
                 ──< Reaction
SearchLog   （記事と非依存：検索ログ）
EditLog     （記事と任意依存：AI編集ログ）
```

**設計上の注目点**

- `ArticleTag.source`：タグがユーザー手動入力（`user`）か AI提案採用（`ai_suggested`）かを記録。AIタグ採用率の計測に使用。
- `Article.bodyRaw` / `Article.bodyFormatted`：原文と整形版を分離して保存。AIが書いた内容と人間が書いた内容を明確に区別する。
- `Article.status`：`published` / `draft` の2値。下書き中の記事はビュー数インクリメントの対象外。
- `SearchLog.wasHelpful`：検索後のフィードバックを保存。北極星指標の計測に使用。
- `EditLog.accepted`：AI編集の採用/却下を記録。AI補助の有効性を検証する。
- `Reaction`：`@@unique([articleId, userId, type])` で二重リアクション防止。

## 4. 意図的に含めなかったもの

| 機能 | 除外理由 |
|------|---------|
| 認証（NextAuth 等） | MVPの優先範囲外。現在はヘッダーのユーザー切り替えで代替。本番化時に追加する。 |
| Slack 通知連携 | 「投稿した感覚が薄い」という動機(M)の問題へのアプローチとして有効だが、外部サービス依存が増えるためフェーズ2以降。 |
| 全文検索エンジン（Elasticsearch 等） | SQLite の `LIKE` 検索で十分な規模。スケール時に移行する。 |
| ストリーミング応答 | AI回答のUX向上に有効だが、実装複雑度に対してMVP段階では優先度が低い。 |
| ベクトルDB（pgvector 等） | 現状の全件コサイン類似度計算は記事数が少ないうちは十分。スケール時に移行する。 |

## 5. 今後の拡張余地

### フェーズ2（短期）

- **認証導入**：NextAuth + メール認証。投稿者・閲覧者の行動を個人単位で追跡できる。
- **AI補助を使った公開完了率の計測**：現状は `EditLog.articleId` で代替計測可能だが、投稿セッション単位での計測基盤が未実装。
- **Slack 連携**：投稿通知・Slack からの検索コマンド。

### フェーズ3（長期）

- **埋め込みモデルの高精度化**：現在の TF-IDF を `text-embedding-3-small` 等へ置き換え。日本語の意味的類似性が大幅に向上する。
- **PostgreSQL + pgvector への移行**：記事数・ユーザー数が増えた時点で SQLite を置き換える。ベクトル近傍探索がDB側に移ることで RAG のスケーラビリティが向上する。
- **パーソナライズ検索**：閲覧履歴・所属チームに基づいた記事レコメンド。
